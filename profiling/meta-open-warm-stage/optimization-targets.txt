Ranked optimization targets:
─────────────────────────────

1. yolofs_find_dirent hash table sizing — 1.43µs self (save ~1.3µs)

   Root cause: YOLOFS_DE_SHIFT=6 → 64 buckets. The meta-open-warm-stage
   workload has 10,000 files in one directory → 156 entries per bucket
   on average. Every lookup walks ~78 entries (half chain on hit),
   doing memcmp per entry. All filenames are the same length (13 chars
   for file-NNNN.dat) so the name_len fast-reject never fires — every
   entry falls through to memcmp.

   Profile confirms: memcmp accounts for 0.49µs and _raw_spin_lock
   for 0.14µs inside yolofs_find_dirent's 1.43µs self time.

   Fix: increase YOLOFS_DE_SHIFT from 6 to 10 (1024 buckets). Average
   chain drops from 156 to ~10 entries. Cost: 8KB per directory that
   has staged entries (vs current 512 bytes). Only allocated on first
   dirent insertion, not for base-only directories.

   Expected improvement: lookup drops from 1.43µs to ~0.1µs.

   Alternative: dynamic resizing (start at 64, rehash when load
   factor > 4). More complex but adapts to any directory size.

   Further optimization: for read-only opens, the dirent lookup result
   is only needed to redirect reads to the staged inode (if the dentry's
   lower_path was set at lookup time to the staged inode, the read
   path could use it directly). But a concurrent COW between lookup
   and open could make the cached lower_path stale, so the dirent
   check is needed for correctness. Caching ino+gen on the dentry's
   d_fsdata (updated on COW) would avoid the hash walk entirely —
   this is a complementary optimization to hash table sizing.

2. yolofs_open_lower → dentry_open — 1.33µs (save ~0.4µs net vs overlayfs)
   yolofs opens a new struct file for the lower fs on every yolofs_open
   via dentry_open → alloc_empty_file. overlayfs uses backing_file_open
   (kernel 6.6+) which is specifically designed for stacking filesystems.
   Fix: switch from dentry_open to backing_file_open.
   Net saving is 1.33 - 0.95 (overlayfs backing_file_open) ≈ 0.4µs.

3. Path walk overhead — 0.23µs
   yolofs has longer absolute paths through .yolofs/mnt/... compared to
   overlayfs's shorter namespace-relative path. This is structural and
   not fixable without changing how yolofs mounts. lookup_fast is now
   at 0.22µs (was 0.91µs before d_revalidate fixes) — essentially at
   parity with overlayfs (0.15µs).

4. [DONE] d_revalidate — was 0.96µs, now 0µs
   Fixed in two steps: RCU-walk fast path (ff83d16), then full removal
   for local lower filesystems (2524942).

Total recoverable: ~1.7µs of 1.90µs gap.
Remaining ~0.2µs: path walk length + overlayfs permission overhead
difference.
